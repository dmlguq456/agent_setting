## capability-system (capability-system)

**structure_summary**: The "capability system" is a portable, runtime-neutral layer under `capabilities/` plus its generation/validation tooling, with four structural parts. (1) Catalog index `capabilities/README.md` (README.md:52 `<!-- GENERATED: harness-manifest.json -->`) whose table (README.md:54-82) lists 30 capabilities with Group (pre/entry/sub/ops), Modes, portable-spec link, portable meaning, and three per-adapter realization paths. (2) 30 per-capability contract files `capabilities/<id>.md`, each a GENERATED-header doc with a fixed skeleton: Contract table (Identifier/Group/Supported modes/Portable meaning/Argument shape/Execution topology/Entry load phase), Invocation Semantics, Artifact Ownership, Role Requirements, Guard Requirements, Adapter Realization (autopilot-code.md:1-139). Entry capabilities add Stage Mapping + Portable Procedure + Mode-Specific Semantics (autopilot-code.md:66-127); sub capabilities are thin (code-execute.md=50 lines, mostly boilerplate). (3) Declarative stage-graph registry `capabilities/topologies.json` (1692 lines) with `recipes[]` keyed by (capability, mode) defining the actual DAG. (4) Media contract `capabilities/report-manifest.schema.json`. Backing tooling: `harness-manifest.json` (repo root, 28KB) is the SoT generating every contract/catalog table; `tools/harness_manifest.py`+`tools/sync-entry-skill-layer.py` render projections; `tools/capability_topology.py` validates topologies.json; `utilities/capability-route.py` compiles/verifies/completes immutable routes from it. topologies.json enums (intensities, execution_topologies, worker_kinds, transports, execution_surfaces, fallback_hops) are the shared vocabulary.

**where_behavior_lives**: Stage graph + dispatch live in `capabilities/topologies.json` recipes. Each recipe: `topology_class`, `direct_predicates`, `promotion_signals`, a `quick` block (topology one-shot-owner, worker_kind capability-owner, write_scope, owner_dispatch_depth=1, max_dispatch_depth=1), a `standard_plus` block with `nodes[]`, plus `completion_gates`/`human_gates`/`resume_retry_boundaries`. Each node declares: `id`, `kind` (worker_kinds enum: capability-owner/pipeline-stage/review-worker/map-worker/resource-runner), `depends_on` (DAG edges), `role` (PORTABLE MODEL ROLE string: deep maker/fast implementer/deep reviewer/fast writer/deep orchestrator/fast reviewer/orchestrator), `inputs`, `outputs`, `write_scope`, `resource_class`, `completion_gate`, `dispatch_depth` (1 or 2), `fallback_hops` (same-harness-headless→cross-harness-headless→native-subagent→inline). Example autopilot-code standard_plus = plan→execute→test→report (topologies.json:224-347). Behavior/role/config declared in THREE layers: (a) portable model role per node in topologies.json `role`; (b) abstract role FAMILY in each contract Role Requirements ("planning role for code-plan", "development role for code-execute", "QA role for code-test" — autopilot-code.md:56-62; code-test.md:61-66 also cites `roles/modes/qa/test.md` by path); (c) concrete bespoke-agent binding in `harness-manifest.json` `capabilities[].requires.roles` (role-team names), resolved to native agents only in adapter SKILL files. Concrete model IDs are explicitly banished ("Concrete model names, subagent frontmatter... belong in adapter files" in every contract).

**duplication_or_drift**: Heavy but GENERATED boilerplate duplication across the 30 contracts (single SoT, low risk). Verbatim-repeated: "Use portable role names from `roles/README.md` and `core/CONVENTIONS.md`. Concrete model names, subagent frontmatter..." in ~28 contracts (autopilot-code.md:54, code-plan.md:33, code-execute.md:28, code-test.md:57, autopilot-spec.md:52, ...). The Guard Requirements 5-bullet list is byte-identical across contracts (autopilot-code.md:85-92, code-plan.md:39-43, code-test.md:71-76). The Adapter Realization + Compatibility Reference "diff -qr byte-identical" paragraph is duplicated ~3x per file × 30 files. topologies.json duplicates the identical 7-item `direct_predicates` array and fallback_hops arrays in all 11 recipes. Real DRIFT seam: node.role free-text strings in topologies.json must stay in sync with core/CONVENTIONS.md:92-99 role table AND capabilities[].requires.roles team names in harness-manifest.json AND roles/README portable_role — there is NO cross-validator tying node.role ↔ capability.requires.roles ↔ roles/README portable_role (capability_topology.py validates node.kind but not node.role), so these can drift independently today.

**shared_vs_surface_specific**: GENUINELY SHARED / already unit-shaped: portable model roles (topologies.json node.role; defined once core/CONVENTIONS.md:92-99), worker_kinds enum, intensity graph, abstract role families in Role Requirements. Critically the capability CONTRACTS (capabilities/*.md) and topologies.json are ALREADY agent-neutral — they never name a bespoke agent (plan-team/dev-team/qa-team...); grep for those names across `capabilities/` returns nothing; they appear only in adapter SKILL files (adapters/claude/skills/*/references/owner-execution.md etc.). The ONE place the capability system binds to bespoke agents is `harness-manifest.json` `capabilities[].requires.roles`, listing the 8 role-teams (dev-team, plan-team, qa-team, editorial-team, design-team, material-team, research-team, external-adversary) mapping 1:1 to `adapters/claude/agents/*.md`. GENUINELY SURFACE/LIFECYCLE-SPECIFIC (stays per-projection): the topology DAG shape itself (dependency edges, resume boundaries, completion gates, write-scopes, human gates, resource-runner detached lifecycle) is capability-level orchestration, not a unit concern — a unit is projected ONTO a node, it does not own the graph. fallback_hops, dispatch_depth, per-node write_scope are execution-surface policy that persists.

**generation_hooks**: Everything in `capabilities/*.md` and `capabilities/README.md` is PROJECTED, not hand-authored: each carries `<!-- GENERATED: harness-manifest.json -->` at its Contract/Catalog table (README.md:52, autopilot-code.md:6). SoT is `harness-manifest.json` (root) consumed by `tools/harness_manifest.py`; `tools/sync-entry-skill-layer.py` and `tools/render-hub.py` render entry-skill/catalog projections; `tools/generated-projections.test.sh`+`tools/entry-skill-layer.test.py` guard drift. topologies.json is validated (not generated) by `tools/capability_topology.py` (node.kind ∈ worker_kinds at :118, execution_surfaces set equality at :184, reviewer write-scope isolation :152, dispatch-depth rules :136, guard-scope/spec-owner :63-72). Routes compiled from it by `utilities/capability-route.py` (imports tools/capability_topology.py; dispatch contract v3; gate_command topologies.json:57). Composition step hooks in at two points: harness-manifest.json capability entries (swap requires.roles→units) and the generators reading them. The model-tier resolution already has its enforced-SoT precedent in `adapters/<a>/config/models.conf` guarded by `tools/check-model-config.py` — the stated mirror for the unit-def SoT+guard pattern.

**migration_implications**: The capability system is scheduled LAST because it is the composition ROOT, not a persona holder. "Referencing bespoke agents" here reduces to ONE field: `harness-manifest.json` `capabilities[].requires.roles` (and its generated catalog projection). Every capability declares it: e.g. autopilot-code requires ['dev-team','editorial-team','plan-team','qa-team'] + caps ['code-plan','code-execute','code-test','code-report','code-refine']; code-execute requires ['dev-team']; code-test ['qa-team']; autopilot-spec ['design-team','material-team','plan-team','qa-team']. Migration turns each requires.roles entry from a role-team name into a UNIT reference; the topology node already carries the two composition seams a unit needs (node.role = portable model role → tier via models.conf; node.kind = worker_kind → which worker-type fragment/lifecycle). A node becomes: pick unit U, project U onto surface S chosen by node.kind + fallback_hops (registered dispatch worker via roles/worker-bootstrap.md+worker-types when headless; native subagent via adapters/<a>/agents when the native-subagent hop is taken). Domain-graded persona-minimization maps cleanly: pipeline-stage/review-worker nodes (code-execute/test/report — mechanical) decompose to near-zero persona and could be generated from node metadata + a thin stage fragment; judgment-heavy capability-owner/deep-maker nodes (code-plan, autopilot-spec research, autopilot-design refs) keep a thin domain core. Because contracts and topologies.json are already agent-neutral, the layer restructure is mostly (a) rewire harness-manifest requires.roles→units, (b) teach generators (tools/harness_manifest.py, sync-entry-skill-layer.py) to emit unit-composed projections, (c) leave topologies.json DAGs intact.

**risks**: Silent role-vocabulary drift: node.role strings (topologies.json), capabilities[].requires.roles (harness-manifest.json), and roles/README portable_role are not cross-validated; capability_topology.py only checks node.kind∈worker_kinds (:118), not node.role. A units migration rewiring requires.roles could leave topology node.role dangling.; Capability contracts are GENERATED from harness-manifest.json; hand-editing capabilities/*.md is a no-op that gets overwritten. Migration MUST route through harness-manifest.json + tools/harness_manifest.py + sync-entry-skill-layer.py or generated-projections.test.sh flags drift.; topologies.json is runtime-load-bearing for utilities/capability-route.py (immutable route compilation, dispatch contract v3, gate_command :57). Restructuring node.role→unit or node.kind semantics risks breaking route compilation/gate completion and the atomic attempt-claim path — the reason it is sequenced LAST.; Domain-grading must key off node.role+kind, not capability group: some 'sub' pipeline-stage nodes still carry judgment (code-plan role 'deep maker'), while capability-owner nodes vary (route-apply, sync). Keying persona-minimization off the group would over-flatten planning capabilities.; resource-runner nodes (autopilot-lab full-run/eval-run) are explicitly NOT agent dispatches (capability_topology.py:129 'resource runner lifecycle is not an agent dispatch'; detached-process transport). Unit-projection must exempt them — no worker/agent surface to compose onto.; The two target surfaces (native subagent adapters/<a>/agents/*.md vs registered dispatch worker roles/worker-bootstrap.md+worker-types/*.md) are selected at runtime by per-node fallback_hops, not statically. A unit projected onto both must keep SEPARATE lifecycles; conflating them violates the CLAUDE.md invariant that native subagents stay distinct from registered headless dispatch.

**open_questions**: Is there (or will there be) a validator tying topologies.json node.role ↔ harness-manifest requires.roles ↔ roles/README portable_role, analogous to check-model-config.py for models.conf? Today none exists.; Under units, does capabilities[].requires.roles become requires.units, and do topology nodes gain an explicit `unit` field, or is the unit inferred from (node.role, node.kind)? The current node has role+kind but no unit id.; How are per-node write_scope/completion_gate/fallback_hops (surface/lifecycle policy) reconciled with a unit's fixed input/output contract — does the unit own an I/O contract that must match node inputs/outputs, and who validates the match?; Do the 'sub' capabilities (code-execute/test/report as their own capabilities/*.md AND as nodes inside autopilot-code standard_plus) collapse into units, causing double-representation (sub-capability contract vs node reference)?; Does external-adversary (roles/README:19 → adapters/claude/agents/codex-review-team.md), which appears in role catalogs but not in any capability requires.roles list, become a unit, and how does its cross-runtime lifecycle project onto the two surfaces?
