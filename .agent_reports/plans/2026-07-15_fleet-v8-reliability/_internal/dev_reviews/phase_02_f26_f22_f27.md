## C1 — 잡 행 kill이 항상 거부됨 · `close_registry_row`가 도달 불가 죽은 코드 (**CRITICAL**)

### 사실

`render.py:1396` 이 잡 엔트리의 identity 절반을 `getattr`로 읽는다:

```python
def _select_entry_job(j, line_idx):
    return {"line": line_idx, "kind": "job", "pid": j.pid,
            "proc_start": getattr(j, "proc_start", None),   # ← 항상 None
```

**`DispatchJob`에는 `proc_start` 필드가 없다.** 실측:

```
DispatchJob has proc_start: False
Session   has proc_start: True
```

`proc_start`를 쓰는 곳은 `procscan.py:331`(**Session 전용**)이 유일하고, `collectors/dispatch.py` 전체에 `proc_start` 문자열이 **0회** 등장한다. 따라서 잡 엔트리의 `proc_start`는 언제나 `None`이고 → `control.py:181`

```python
if not proc_start:
    return False                      # no identity half → cannot prove the target
```

→ `verify_target` 항상 False → `kill_target`이 `control.py:217`에서 **무조건 거부**한다.

실측(`os.kill` 모킹, 시그널 0회):

```
job entry proc_start -> None
kill_target verdict: refused | os.kill called: False
logged reason: start_time_mismatch
```

### 파급 3건

1. **spec 계약 미이행**: prd.md:251 "기본 허용 대상 = … **registry 잡의 exact pid**" · 계획 §6.4-4 동일. 잡 행은 `render.py:1788 if job.pid:`로 **선택은 되고**(`dispatch.py:1016`/`:952`가 pid를 backfill한다), 커서도 가고, 프롬프트도 뜨고, `y`도 받는다 — 그리고 **항상 조용히 거부된다**. 사용자에게는 "kill이 안 먹는 행"으로 보인다.
2. **prd.md:255 예외가 실현되지 않음**: `_close_job_row_if_registry`는 `render.py:2348 if r == "ok":` 뒤에만 호출된다. `r`이 "ok"가 될 수 없으므로 **`close_registry_row`는 프로덕션에서 단 한 번도 실행되지 않는다.** 무write 불변식의 유일한 예외로 설계·논증·테스트된 서브시스템 전체가 죽은 코드다.
3. **거부 사유가 사실과 다름**: `reason="start_time_mismatch"`는 "start-time이 **달랐다**"는 뜻이지만 실제로는 "start-time을 **수집한 적이 없다**"이다. action log가 감사 근거인데(prd.md:253) 이 로그를 읽는 사람은 PID 재사용이 실제로 탐지된 것으로 오독한다.

### 왜 테스트가 못 잡았나 (구조적 지적)

`TestRegistryCloseParity`(6건)와 `RegistryCloseTest`(8건)는 `control.close_registry_row(...)`를 **직접 호출**한다. 함수 자체는 정확하다 — 다만 **아무도 그것을 부르지 않는다**. 계획 §9의 "registry 마감 동형 검증" 게이트가 초록인 이유가 이것이다. `_do_kill` → `_close_job_row_if_registry` → `close_registry_row` **통합 경로를 타는 테스트가 0건**이다.

### 수정 방향 (택1 — 구현은 개발팀 소관)

- **(a) 완성**: `DispatchJob`에 `proc_start` 필드를 additive로 추가하고, pid를 backfill하는 두 지점(`dispatch.py:952`, `:1016`)에서 `read_proc_start(pid)`를 같이 채운다. 그 뒤 `_do_kill`→`close_registry_row` 통합 테스트 1건 추가. → spec 계약 충족.
- **(b) 명시적 descope**: 잡 행을 selectable에서 제외(`render.py:1788`)하고 `close_registry_row`/parity 스위트를 함께 내리고, prd.md:251/255를 후속 obligation으로 기록. → 죽은 코드 + 거짓 초록 제거.
- 어느 쪽이든 `verify_target`이 `proc_start` 부재와 불일치를 **구분해 로그**하도록 사유를 분리할 것(`no_proc_start` vs `start_time_mismatch`).
- **(a)를 택할 경우 주의**: `verify_target` 자체는 손대지 말 것. `not proc_start → False`의 fail-closed가 바로 이 결함을 **안전하게** 막아준 가드다.

## M2 — `control.py`가 자기 자격 화이트리스트를 강제하지 않음 (MINOR)

`control.py`의 모듈 docstring은 "Everything here is fail-closed"라 선언하고, 존재 이유를 "curses 없이 테스트 가능한 가드"라 못박는다. 그런데 **자격 판정의 절반이 render에만 있다**.

1. **`SINGLE_CONFIRM_STATES`(control.py:37)는 죽은 상수다.** 전체 트리에서 참조가 정의(:37)와 docstring(:197) 2곳뿐 — 실행 경로에서 **한 번도 읽히지 않는다**. docstring `"'single' → SIGTERM, allowed for SINGLE_CONFIRM_STATES"`는 강제되지 않는 주장이다. 실제 구현은 화이트리스트(기본 거부)가 아니라 **블랙리스트(기본 허용)** — `DOUBLE_CONFIRM_STATES`에 없는 모든 상태가 단일 확인으로 통과한다.
2. **`kill_target`의 이중확인 재검사가 caller보다 약하다.** `control.py:212`:

```python
if requires_double_confirm(state) and approval == "single":   # ← registry_status 누락
```

`render.py:2318`은 `requires_double_confirm(entry.get("state"), entry.get("status"))`로 **status 축까지** 넘기는데, control 자신의 belt-and-suspenders는 그 축을 버린다. `state="idle", status="busy"` 대상은 control 단독으로는 단일 확인을 통과한다. `kill_target` 시그니처에 `registry_status`가 없어 **강제할 방법 자체가 없다**.

지금은 안전하다 — `render._selectable_session`(render.py:1366-1383)이 평범한 idle을 애초에 선택 불가로 막고(테스트로 고정: test:438-443), busy 축도 render가 처리한다. **현재 실제 위험은 0.** 다만 "테스트 가능한 단일 안전 모듈"이라는 이 파일의 존재 이유에 비춰, 자격 등급이 curses 레이어에 흩어져 있는 것은 설계 의도와 어긋난다.

- 수정: `kill_target(..., registry_status=None)`을 받아 `state not in SINGLE_CONFIRM_STATES and approval == "single"` → 거부(기본 거부)로 뒤집고, `requires_double_confirm(state, registry_status)`를 완전 인자로 호출.

## M3 — 커서가 index 기반이라 리빌드 중 조용히 재조준된다 (MINOR)

`_CURSOR`(render.py:2158)는 주석대로 "index into `_SELECTABLE`, not a screen row"다 — 그런데 **identity가 아니라 순번**이다. 보드가 리빌드되면 같은 index가 다른 세션을 가리킨다. 실측 재현:

```
tick1 targets: ['ghost-A', 'ghost-B'] | cursor=0 -> ghost-A
tick2 targets: ['ghost-B', 'ghost-Z'] | cursor=0 -> ghost-B     ← ghost-A 소멸
>>> 사용자는 ghost-A를 겨눴는데 프롬프트는 ghost-B를 지목
```

**다만 브리프가 우려한 "엉뚱한 행을 죽인다"는 성립하지 않는다.** 3중으로 막힌다:

1. `_SELECTABLE`은 `_build_lines` **안에서만** 변이하고(:1423 리셋, :1789/:1847 append), `_build_lines`는 `_draw`에서만 불린다. `_draw`는 루프와 **동기적**이라 "화면은 그대로인데 타겟 맵만 바뀌는" 창은 **존재하지 않는다.** 프롬프트 대상 = **마지막으로 화면에 칠해진 하이라이트 행**이 항상 보장된다.
2. `_PROMPT["entry"]`는 스냅샷 dict라 프롬프트가 뜬 뒤 리빌드가 대상을 못 바꾼다.
3. `_prompt_segs`가 대상을 명시 지목한다 — `kill ghost-B [pid 90002] unused?`. 게다가 `kill_target`이 그 엔트리의 `proc_start`로 재검증하므로 **프롬프트가 지목한 그 프로세스 외에는 절대 죽지 않는다.**

즉 잔여 위험은 "사용자의 조준과 보드의 이동이 어긋난 채 확인을 눌렀다"는 **지각 층**이고, 프롬프트를 읽으면 잡힌다. 그래도 index가 아니라 `pid`로 커서를 고정하고 리빌드마다 index를 재탐색하면 이 클래스가 통째로 사라진다(대상 소멸 시 선택 해제).

- 같은 뿌리의 부수 지적: `entry["state"]`도 빌드 시점 스냅샷이다. `unused` → `working` 전이가 draw와 confirm 사이에 끼면 **낡은 state로 확인 등급이 매겨진다**(단일 확인으로 live 세션 종료). 창이 sub-second라 실사고 확률은 낮지만, `_do_kill` 직전에 등급을 재평가하면 함께 닫힌다.
- 커버리지: `test_cursor_moves_and_clamps`(test:489)는 clamp만 본다. **리빌드를 가로지르는 커서 동일성 테스트가 0건**이다.

### G2 — PID 재사용 가드 / TOCTOU: 실질 위험 무시 가능

`verify_target`(:217) → `os.kill`(:225) 사이 창은 마이크로초다. 오시그널이 성립하려면 그 창 안에서 대상이 종료되고 **+ pid가 재할당**돼야 하는데, Linux pid는 순차 할당이라 `pid_max`(≈32768) 만큼의 프로세스 생성이 그 사이에 일어나야 한다. **실질 0.**

완전 제거는 `os.pidfd_open`(**Python 3.9+**)이 필요한데 본 프로젝트는 **3.8.10**이라 사용 불가 — 즉 현 런타임에서 이 구현이 **도달 가능한 최선**이다. 가드 순서도 계획 §6.4대로다: approval 유효성 → `is_excluded` → 이중확인 → `verify_target` → kill. **비용이 비대칭이라 fail-closed가 옳다**는 docstring(:174-175)의 논거에 동의한다.

### G3 — `is_excluded`: 합성 결과는 fail-closed

브리프 질문 3. `_ancestors`(:101)는 `/proc` 실패 시 `break` 후 부분 리스트를 반환 → **그 축만 놓고 보면 fail-open**이다. 그러나:

- `/proc` 전면 불가(컨테이너 등) 시 `verify_target` → `read_proc_start` → `None` → **kill 자체가 거부**된다. 동일한 `/proc` 의존성이 뒤에서 fail-closed로 받는다. **합성 결과는 안전.**
- 부분 실패도 사실상 불가: `/proc/<pid>/stat`은 **world-readable**이라 타 유저 프로세스도 읽힌다(실측: root 소유 pid 1의 stat을 정상 판독). 즉 fail-open 분기는 현실에서 도달하지 않는다.
- 커버 범위: 자기 자신·`getppid()`·**전체 조상 계보**(tmux 서버·터미널 포함 — tmux pane에서 fleet의 조상 사슬에 tmux 서버가 들어온다)·`CLAUDE_CODE_SESSION_ID`로 해석한 구동 세션·`pid<=1`. `int()` 실패도 `return True`(:153)로 **입력 불명 = 제외**. `_live_targets`(:2172)가 **프롬프트 이전 단계**에서 거르는 것도 계획 §6.4-3 그대로고, 거기서도 예외를 `continue`로 삼켜 fail-closed다(:2184-2185).
- `CLAUDE_CODE_SESSION_ID`는 **실재하는 env**임을 확인했다(현 셸에 존재, `dispatch-headless.py:88`도 사용). 자식에게 상속되므로 fleet이 세션 안에서 뜨면 구동 세션이 정확히 배제된다.

**다만 이 안전은 "`_ancestors`의 fail-open을 `verify_target`이 받아준다"는 암묵적 결합에 의존한다** — 어느 쪽 docstring에도 안 적혀 있다. 주석 1줄 값어치는 있다.
