# fleet v10 — 처리 과정 시각화 완료 보고

**결론: 목표대로 착륙했습니다.** fleet이 이제 파이프라인의 **실제 실행 계획서(route record)를 읽어서** 처리 과정을 그립니다. 추측이 아니라 기록된 사실입니다. 테스트 468 → **519, 회귀 0**.

다만 이 사이클의 진짜 성과는 기능이 아니라 **간판 기능이 조용히 죽어 있던 것을 잡아낸 것**입니다. 그리고 보고에 반드시 필요한 **인프라 문제 2건**을 별도로 발견했습니다(§5) — 그쪽이 더 급합니다.

---

## 1. 무엇이 생겼나

### 처리 과정 뷰 (새 화면)

`p` 키를 누르면 프로젝트 그룹 대신 **파이프라인 중심**으로 화면이 재편성됩니다. (비대화식으로는 `--view process` 또는 `FLEET_VIEW=process`.)

```
▸ [code·dev·standard] rt-1120bb39 — 4/4 nodes                    ← 다 끝난 파이프라인은 1행으로 접힘
▾ [code·dev·standard] rt-27f7bc9f — 0/4 nodes ⚠ failed node      ← 실패는 자동으로 펼쳐짐
  plan ✕ › execute ○ › test ○ › report ○
▾ [lab·eval·standard] rt-6f5423d0 — 1/6 nodes ⏳10m ⚠ failed node
  setup ✓10m
    ├ eval-asr ● 9m (Sonnet 5·medium)                            ← 병렬 분사가 세로로 갈라짐
    ├ eval-sep ✕ 3m
    └ eval-vad ○
  › aggregate ○ › report ○                                       ← 다시 합류(fan-in)
   └▸🚀 fleet-v10-execute  claude  Opus 4.8  ⏳8m                 ← 그 노드를 실제로 도는 세션
      └⚡ explore ⏳2m                                             ← 그 세션이 띄운 서브에이전트
```

- **`✓` 완료 / `●` 실행 중 / `○` 미기동 / `✕` 실패** — 색뿐 아니라 글리프+텍스트로 이중 표기(색맹 판독 가능).
- **분사(fan-out)와 합류(fan-in)가 그려집니다.** 병렬 노드는 들여쓴 세로 분기, 다시 모이는 지점은 `›`로 합류.
- **서브에이전트까지 이어집니다** — 노드 → 그 노드를 도는 세션 → 그 세션의 서브에이전트가 한 화면에 중첩.
- 마우스로 접고 펼칩니다. 기존 kill 조작 문법은 그대로 살아 있고, **kill 경로는 새로 만들지 않았습니다**(기존 확인 프롬프트 한 곳으로 합류).
- 60/120/168열 전 폭에서 넘침 0. 좁아지면 과거 노드부터 접고 **지금 도는 노드를 마지막까지 살립니다**.

이것이 오래 말씀하신 **"분사·서브에이전트 처리 과정이 보이는 화면"**의 첫 실물입니다.

### 계획서 기반 진행 표시 (breadcrumb)

기존에는 `plan › exec › test` 3단이 **하드코딩**돼 있었습니다. 실제 파이프라인은 4단(`plan → execute → test → report`)이고, 하드코딩 표에 아예 없는 노드도 실재했습니다(예: 스펙 파이프라인의 `research`). 이제 record가 있으면 **record의 진짜 노드 이름과 순서**를 씁니다. record가 없는 작업은 기존 표시 그대로 — 회귀 0.

**중요한 설계 원칙**: record는 **레일만 깔고, 점등은 실측이 합니다.** record에 `execute`가 있어도 실제로 도는 자식이 없으면 그 노드는 켜지지 않습니다. 계획서가 현실을 덮어쓰지 못하게 한 것입니다.

### 모델 워커 사용량 1행

`⚙ governor 2/5` — 동시 모델 워커 lease 현황이 pulse 옆에 붙습니다. 절반 미만이면 숨깁니다(건강하면 조용히).

**까다로운 부분을 정확히 처리했습니다**: fleet은 이 파일을 절대 쓰지 않으므로 **이미 죽은 lease가 파일에 남아 있을 수 있습니다.** 그대로 세면 `5/5`라고 거짓말합니다. 그래서 표시 직전에 메모리에서 죽은 lease를 걸러내고, **PID 재사용까지 배제**(프로세스 시작시각 대조)합니다. 파일은 손대지 않습니다.

## 2. 이 사이클의 핵심 서사 — 간판 기능이 자기 자신 앞에서 죽어 있었다

검증 1회차가 잡아낸 것입니다. 화면은 이랬습니다:

```
▾ [code·dev] fleet-v10-process-view — no route record
  plan › exec › test
```

**이 사이클 자신의 record가 "record 없음"으로 떨어지고 있었습니다.** 그런데 record는 멀쩡했습니다 — 파일도 있고, 해시 재계산도 일치하고, 손으로 불러보면 4개 노드가 정상적으로 나왔습니다.

**근인은 필드 하나였습니다.** 끝난 노드의 증거를 보존하는 스캔 패스가 `route_file`(record가 어디 있는지)을 안 실었습니다. 그래서 그 route의 노드가 전부 끝나 있으면 **record 경로를 아무도 모르게 됩니다.** 증거는 보존됐는데 고아가 된 것입니다.

**여기가 무서운 지점입니다.** 이 기능은 "record를 못 읽으면 조용히 기존 표시로 내려간다"는 tolerant 설계입니다. 그 관대함이 **기능 결손을 정확히 가려주고 있었습니다.** 화면은 멀쩡해 보이고, 테스트는 통과하고, 기능만 죽어 있었습니다. 검증이 "부재도 파싱 실패도 해시 불일치도 아닌데 왜 fallback이 일어나는가"를 끝까지 물고 늘어져서 잡혔습니다.

**게다가 테스트가 이 결함을 못 잡은 이유도 드러났습니다** — 완료 route 테스트가 **프로덕션이 만들 수 없는 상태**를 주입하고 있었습니다(끝난 노드의 잡은 실전에서 목록에 남지 않습니다). 테스트 하네스 안에서만 존재하는 통과였습니다. 실제 형태로 재작성했습니다.

**그리고 같은 버그가 두 번째 위치에 또 있었습니다.** 수정 후 검증 2회차가 발견 — conductor가 자기 record 카드 **2행 아래**에 "no route record"로 **한 번 더** 표시됐습니다. 한 파이프라인을 두 번, 서로 모순되게 진술한 것입니다. 근인은 동일: 또 다른 코드가 "살아 있는 잡만 보면 된다"는 **똑같은 틀린 가정**을 쓰고 있었습니다. 수정이 절반만 적용됐던 것입니다.

> 참고로 구현 스테이지는 이 중복을 "기존 동작, 허용 가능"이라고 진단했고, **검증이 그것을 반증했습니다.** 다만 구현 쪽이 스스로 flag해서 판정을 요청했기 때문에 드러났습니다 — 은폐는 없었습니다.

**두 결함 모두 live 실측으로 수정을 확인했습니다.** 검증 1회차에 heuristic(추측 표시) 5개 → 2회차 **0개**. 중복 카드 0회.

**수정이 진짜인지도 확인했습니다** — 복제본에서 수정을 되돌려보는 mutation 테스트로, 새 회귀 테스트가 **실제로 실패하는지** 검증했습니다. 되돌려도 통과하는 "공허한 테스트" 2건도 이 과정에서 발견해 보강했습니다.

## 3. 정직한 결손 — 숨기지 않은 것

**이 사이클이 구현하지 **않기로** 결정한 것들이고, 그 결정이 성과입니다.**

### completion gate 통과 여부는 표시하지 않습니다

spec은 "gate 통과 여부"를 요구했습니다. 구현하지 않았습니다. **통과를 증명하는 증거가 이 시스템에 존재하지 않기 때문입니다.**

- `completion_gate=` 필드는 **작업을 띄우는 시점에 박히는 선언**입니다. 실측 반증 2건: **시작도 안 한 노드**가 이 필드를 갖고 있고, **죽은 노드**도 갖고 있습니다. 이걸 "통과"로 읽으면 둘 다 ✓가 됩니다.
- `status=done`조차 성공이 아닙니다 — fleet이 kill한 작업도 `done`으로 마감됩니다.
- 진짜 통과 마커를 쓸 수 있는 도구는 존재하지만, **repo 전역에 그 마커 파일이 0건**입니다. 아무도 호출하지 않습니다.

없는 근거로 ✓를 그리는 것은 이 기능의 존재 이유("정책 따로, 표시 따로"의 불일치 제거)를 정면으로 뒤집는 일입니다. **그래서 `—`로 남겼습니다.** `gate_passed` 같은 키는 어디에도 만들지 않았습니다(전 페이로드 부재 확인).

**사용자 판단 사항입니다**: 이 결손을 메우려면 dispatch 파이프라인이 노드 인계 시점에 실제 통과 마커를 쓰도록 stage-dispatch 규약이 바뀌어야 합니다. 그렇게 되면 지금 구조에 **덧붙이기만 하면 됩니다**(재작업 불필요 — 자리를 비워뒀습니다).

### 분리된 실행 registry 관측은 스킵 + 이월

spec은 "소스가 없으면 정직하게 스킵"을 예상했습니다. **실제 판정은 더 좁고 정확합니다** — 소스(도구)는 **실재합니다**. 다만 그 도구는 registry 경로를 **호출자가 매번 넘기는 인자**로 받고 기본 경로가 없습니다. 즉 fleet(외부 관찰자)이 **찾아갈 canonical 경로가 없습니다.**

**"없다"가 아니라 "찾을 수 없다"** — 이 구분을 이월 기록에 남겼습니다. 추측 경로를 뒤지는 것은 "추측 표시 금지" 원칙 위반이라 하지 않았습니다.

### 그 외 이월

- `tracked_gate_evidence` 미구현 (소스 참조 0건) — `a` 상세는 게이트 **이름**만 dim 표시.
- 디자인 지적 2건: 결손 카드의 시각 무게(🟡, 중복 카드 해소로 체감 완화), footer 접기 힌트 부재(🟢).

## 5. ★ 별건 — 보고해야 할 인프라 문제 2건

**이 사이클 산출물이 아니라 수행 중 발견한 것입니다. 기능보다 이쪽이 급합니다.**

### 5.1 dispatch 브로커가 모든 프로젝트를 줄세우고 있습니다 (head-of-line blocking)

`dispatch-broker.py:510`이 **전역 뮤텍스를 쥔 채로** 워커를 **동기 실행**합니다:

```python
with self.request_guard:
    subprocess.run(...)      # ← 자식이 끝날 때까지 락을 놓지 않는다
```

락이 자식의 **전 생애 동안** 잡혀 있습니다. 워커 하나가 12분을 돌면 **그동안 모든 프로젝트의 dispatch가 12분 멈춥니다.**

**실측**: 이 사이클 도중 **타 프로젝트 워커 1개**가 약 **12분간 전 fabric을 차단**했고, 대기 스레드가 **6개** 쌓였습니다. 브로커의 요청 접수와 워커 실행이 같은 락을 공유하는 것이 원인으로, 구조적으로 **동시성이 1**입니다.

### 5.2 route record가 불변인데 브로커 인스턴스를 고정합니다

route record는 **immutable**입니다. 그런데 그 안에 **mutable한 `broker_instance`**가 박힙니다. 브로커가 롤오버하면:

- client는 env에 바인딩된 인스턴스를 보고,
- broker는 envelope을 **각각 살아 있는 인스턴스**와 대조하고,
- → 그 route의 ordinal-1 hop은 **영구히 불가능**해집니다.

**이 사이클이 직접 그 경로를 밟았습니다.** plan만 ordinal-1(브로커 경유)로 실행됐고, execute/test/report는 record가 명시한 checked 순서를 따라 **ordinal-3 native-subagent**로 내려갔습니다(`fleet_visibility=degraded`).

계약 이탈은 아닙니다 — record가 예비한 폴백의 정상 소비입니다. 다만 **불변 기록에 가변 식별자를 박는 설계가 시간이 지나면 route를 자동으로 열화시킨다**는 뜻이고, 아이러니하게도 **이번 사이클은 route record를 보이게 만드는 작업이었으면서 정작 자기 record의 1급 경로를 못 쓴 채 수행됐습니다.**

## 7. 판단이 필요한 것

1. **`--view` 플래그를 남길지** — spec은 `p` 키만 확정했습니다. `--view`는 비대화식 검증(3폭 캡처 + 디자인 비평)을 **수행 가능하게 만들기 위해** 추가한 투영이고, `p`와 **같은 전역 하나**를 씁니다. spec 확장이 아니라 구현 세부로 판단했습니다. 반대하시면 `FLEET_VIEW=process` 환경변수만 남기고 CLI 표면을 0으로 줄일 수 있습니다(동일 진입점, 이미 구현돼 있음).
2. **completion gate 통과 표시** — 지금은 결손(`—`)입니다. 메우려면 dispatch 파이프라인이 실제 통과 마커를 쓰도록 규약이 바뀌어야 합니다. §3 참조.
3. **브로커 head-of-line blocking** (§5.1) — 이건 fleet 사이클과 무관하게 **모든 프로젝트에 지금 영향을 주고 있습니다.** 우선순위 판단이 필요합니다.
