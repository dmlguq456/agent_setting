> **Front-layer map.** This document explains how the whole harness fits
> together along four axes. Precise definitions live in the source documents

> overall flow remains visible without duplicating contracts.

## 0. The 30-second map

This repository is a portable agent harness built from instructions and local
tools. It is **not a setup for Claude Code, Codex, OpenCode, or any other single
runtime**. A main agent reads the current context—working directory, artifacts,
and the user's request—then selects an appropriate capability pipeline. Each
runtime adapter projects that shared contract onto the native surfaces the
runtime actually supports.

Artifacts accumulate under `.agent_reports/`; existing `.claude_reports/`
directories remain readable as a legacy alias. Deterministic hooks enforce
mechanical ordering and safety boundaries. Loops inspect and maintain the
system outside interactive sessions. Semantic decisions, including whether
past memory is relevant, remain the acting agent's responsibility.

Communicate naturally in the language you use. User-facing output follows the
conversation language unless you specify a different audience, publication
venue, or artifact language.

## ① Workflows — from request to artifact

Describe the desired outcome in natural language. The main agent assembles the
relevant options and runs the matching pipeline. Hard ordering gates prevent
later stages from silently skipping required foundations:
`research/analyze → spec → code`; document work follows
`research/analyze → draft → refine → apply`.

Four common tracks:

```text
📄 Documents   analyze-project / autopilot-research → autopilot-draft → autopilot-refine ↻ → autopilot-apply
🔬 Research    analyze / autopilot-research → autopilot-spec ↻ → autopilot-code ↻ → autopilot-lab ↻
💻 Apps        autopilot-spec ↻ → autopilot-design → autopilot-code ↻ → autopilot-ship ↻
📦 Libraries   analyze-project → autopilot-spec ↻ → autopilot-code ↻
```

Invoke each stage with one natural-language request, such as “Turn these
results into a report.” The `↻` symbol means that review and correction may
repeat.

- **Artifacts:** pipelines write into domain folders below the artifact root,
  including `research/`, `analysis_project/`, `documents/`, `spec/`, `plans/`,
  and `experiments/`. Code work separates the current blueprint in `spec/`
  from individual work cycles in `plans/<date>_<slug>/`.
- **Follow-up changes:** in a spec-backed project, first inspect the existing
  artifacts and check spec drift, then route implementation through
  `autopilot-code` instead of applying an ungrounded edit.

See [`core/WORKFLOW.md`](core/WORKFLOW.md) for routing rules and
[`roles/response-policy.md`](roles/response-policy.md) for response behavior.

## ② System structure — parts and relationships

```text
natural-language request
          │  main agent routes the work
          ▼
capabilities/ (27) ──invoke──► roles/ (8) ◄── role modes (roles/modes/)
          │                    │
          │                    └ runtime-native agents generated by adapters
          ▼
.agent_reports/ ◄──guarded── hooks/ (artifact order · git state · spec gates)
       │
       └────────────────────► memory/ (shared mem.py DB store)
                                     ▲
outside sessions: loops/ ──inspect and maintain──┘
```

- **Capabilities** are verbs: work entrypoints and pipelines such as
  `autopilot-code`, `autopilot-research`, `audit`, and `post-it`.
- **Roles** define portable responsibilities such as planning, development,
  QA, research, material gathering, design, and editing. Adapters map them to
  runtime-native agent surfaces where supported.
- **Hooks** are deterministic guards. They enforce mechanical invariants such
  as artifact order, git safety, spec grounding, and valid memory write paths;
  they do not replace semantic agent judgment.
- **Memory** stores working, durable, and profile records in one database.
  Retrieval is agent-initiated, while adapters may expose bounded lifecycle
  context where their runtime supports it.
- **Loops** describe when recurring inspection and maintenance run outside the
  interactive capability pipeline.

See [capabilities/README.md](capabilities/README.md),
[roles/README.md](roles/README.md), and
[`core/CONVENTIONS.md`](core/CONVENTIONS.md).

## ③ Operations — loops outside sessions

Capabilities, roles, and hooks govern work inside a session. Loops operate
outside it. They do not replace pipeline work; they schedule, inspect, clean
up, and verify the surrounding system.

| Loop | Kind | Typical trigger | Responsibility |
|---|---|---|---|
| **oncall** | scheduled | cron 05:37 | Inspect repositories, artifacts, loops, and dispatch jobs; report findings |
| **note** | scheduled | cron 05:03 | Route the previous day's artifacts into worklog-board notes |
| **study** | scheduled | Sunday 06:17 | Compare external developments with the current harness and propose improvements |
| **drill** | event-driven | after behavioral contract changes | Exercise fixture scenarios and score behavioral regressions |

Loop autonomy follows one boundary: reversible, mechanically obvious actions
may run unattended and be reported; decisions requiring interpretation remain
for agent or user review. Drill is never run automatically during ordinary
maintenance because it can launch headless runtime sessions and spend tokens.

See [`loops/README.md`](loops/README.md).

## ④ Principles → implementation

| Principle | Source | Implementation mechanism |
|---|---|---|
| **Runtime-neutral substrate** | DESIGN §0 | Portable capabilities and roles projected onto runtime-native surfaces |
| **Determinism for mechanical invariants** | DESIGN §0.5 | Artifact, git-state, spec, and memory-write guards |
| **Agent-owned semantic judgment** | MEMORY D-40 | No fixed recall signal words or prompt classifier; the acting agent chooses retrieval |
| **Meaning ↔ rule boundary checks** | DESIGN §0.7 | `autopilot-spec`, audit, and targeted drill regressions |
| **Hard ordering gates** | WORKFLOW §0 | `hooks/artifact-guard.sh` |
| **Safe git state** | OPERATIONS §5.9 | `hooks/git-state-guard.sh` blocks unsafe edits during merge/rebase/detached states |
| **Isolated parallel dispatch** | OPERATIONS §5.10 | `<repo>-wt/<slug>`, `.dispatch/jobs.log`, and liveness checks |
| **Unified memory lifecycle** | DESIGN §7 / MEMORY §7 | `mem.py` DB plus adapter lifecycle bridges |
| **Behavioral self-tests** | loops §L4 | `loops/drill/` fixtures, assertions, and token accounting |

See [`core/DESIGN_PRINCIPLES.md`](core/DESIGN_PRINCIPLES.md).
